## What is Firecracker and Why Does AgentCore Use It?

Before we can understand AgentCore's session model, we need to understand the virtualization technology that powers it.

**Firecracker** is an open-source virtualization technology developed by AWS that enables you to create and manage secure, fast microVMs. Unlike traditional virtual machines that carry significant overhead, Firecracker is purpose-built for running workloads that need strong security isolation without sacrificing speed or resource efficiency.

Firecracker's key properties make it uniquely suited for serverless and agent workloads:

- **Fast boot time**: Firecracker VMs can boot in under 125 milliseconds, making them practical for on-demand workloads
- **Strong isolation**: Each microVM has its own kernel, providing hardware-level isolation between workloads
- **Minimal overhead**: The Firecracker VMM (Virtual Machine Monitor) itself has a tiny footprint, enabling thousands of VMs to run on a single host
- **Security by design**: A minimal device model reduces the attack surface compared to traditional hypervisors

You may already be familiar with Firecracker without realizing it. AWS Lambda and AWS Fargate both use Firecracker under the hood to provide the fast, secure, isolated execution environments that millions of customers rely on every day. AWS Bedrock AgentCore follows the same pattern, using Firecracker to give each agent session its own dedicated, isolated execution environment.

This isolation model is particularly important for AI agents. Agent workloads often involve sensitive data, external tool calls, and stateful multi-step reasoning. Running each session in a dedicated microVM ensures that one customer's agent cannot interfere with another's, that sensitive data does not leak between sessions, and that a misbehaving agent cannot consume resources belonging to other workloads.

## AgentCore VM Architecture: The Four Layers

To understand how sessions and VMs relate to each other, it helps to think about AgentCore's architecture as four distinct layers stacked on top of each other.

### Layer 1: The Control Plane

At the top sits the AgentCore Control Plane. This is the platform-managed layer that you interact with when you make API calls to AgentCore. The Control Plane is responsible for:

- **Authentication and Routing**: Validating your AWS credentials (SigV4 or OAuth 2.0) and routing your request to the correct destination
- **Resource Allocation**: Finding available capacity and reserving compute resources for new sessions
- **Session Management**: Tracking which session IDs map to which running VMs, and enforcing the 15-minute inactivity timeout
- **Load Balancing**: Distributing requests across the available infrastructure
- **Monitoring and Metrics**: Collecting telemetry data and publishing it to CloudWatch

The Control Plane is fully managed by AWS. You do not need to configure or operate it. However, understanding what it does is essential because it is the component that makes the cold start versus warm start decision on every single request.

### Layer 2: The Session Layer

Below the Control Plane sits the Session Layer. This is where the critical mapping between session IDs and Firecracker VMs lives. Each active session corresponds to exactly one running Firecracker microVM. This is the most important architectural fact in this entire post:

> **One Session ID = One Firecracker MicroVM**

When the Control Plane receives a request, it looks up the session ID in its session registry. If it finds a running VM associated with that session ID, it routes the request directly to that VM. If it does not find one — because the session is new, expired, or was never created — it allocates a new VM. That allocation process is what we call a cold start.

Each session has a Time-To-Live (TTL) of 15 minutes of inactivity. If no requests arrive for a session within that 15-minute window, the Control Plane terminates the associated VM and removes the session from its registry. The next request using that session ID will trigger a new cold start.

### Layer 3: The Firecracker MicroVM Layer

Each active session gets its own dedicated Firecracker microVM with the following resource allocation:

| Resource | Allocation |
|----------|-----------|
| vCPU | 2 cores |
| RAM | 8 GB (hard limit) |
| Network | Isolated namespace |
| Filesystem | Dedicated volume |
| Session TTL | 15 minutes of inactivity |

The VM provides complete isolation from all other sessions. Its network namespace is separate, its filesystem is separate, and its memory is separate. This means that even if you are running hundreds of concurrent AgentCore sessions, each one operates as if it has its own private server.

### Layer 4: The Container Layer

Inside each Firecracker microVM runs a Docker container. This container is pulled from Amazon ECR and contains your agent application code. The container runs a Python 3.11 runtime, and inside that runtime lives your LangGraph agent, your LangChain dependencies, your Bedrock SDK integrations, and any other code you have packaged into your container image.

This layered model — Control Plane → Session → MicroVM → Container → Python Runtime → Your Code — is what you need to keep in mind when thinking about cold start latency. Every layer in this stack needs to be initialized during a cold start, and each one contributes to the total time your user waits for that first response.

## Session IDs: The Key to Everything

Now that you understand the architecture, let us talk about session IDs in detail, because they are the single most important concept for controlling AgentCore performance.

### What is a Session ID?

A session ID is a UUID-format string that serves as the unique identifier for a Firecracker VM instance. It looks like this:

```
a3f8c2d1-7b4e-4f9a-8c3d-2e1f5a6b7c8d
```

When you include a session ID in your AgentCore API request, you are telling the Control Plane: "I want to use the VM associated with this identifier." The Control Plane uses this as a routing key to find the right VM and send your request to it.

### Three Ways to Provide a Session ID

There are three ways a session ID enters the picture:

**Option 1: Let AgentCore create one for you**

If you make a request without providing a session ID, AgentCore creates a new UUID, spins up a new VM (cold start), and returns the session ID in the response. You must capture this session ID and reuse it for subsequent requests if you want warm invocations.

**Option 2: Provide your own session ID**

You can generate a UUID yourself and include it in your request. If AgentCore has never seen this ID before, it triggers a cold start. If it has seen it and the session is still active, it routes to the existing warm VM.

**Option 3: Use the SessionManager pattern**

For production applications and performance testing, we recommend using a SessionManager class that handles session creation, tracking, and lifecycle management for you. We will show you the full implementation later in this post.

### The Golden Rule

Here is the rule that determines whether every single AgentCore request is fast or slow:

> **New Session ID → Cold Start (~40 seconds)**
> **Existing Session ID (within 15 min) → Warm Start (~3-5 seconds)**

This sounds simple, but it is surprisingly easy to get wrong. We have seen production applications where developers were generating a new UUID for every request, effectively making every single invocation a cold start. The application worked correctly but was 10-15x slower than it needed to be, and the root cause was a single line of code that created a new session ID instead of reusing an existing one.

### An Important Difference from AWS Lambda

If you are coming from an AWS Lambda background, you might expect warm state to be automatic. In Lambda, the platform manages container reuse transparently — you do not need to do anything special to get warm invocations. AgentCore works differently.

| Aspect | AWS Lambda | AWS Bedrock AgentCore |
|--------|-----------|----------------------|
| Warm State | Implicit and automatic | Explicit — requires session ID reuse |
| Developer Control | None needed | Full control required |
| Session Tracking | Not needed | Required for warm starts |
| Isolation Model | Container reuse | Dedicated MicroVM per session |
| State Persistence | Limited | Full session state maintained |
| Timeout | Configurable | 15 minutes of inactivity |
| Warm Invocation Time | Milliseconds range | 2-8 seconds |
| Cold Start Time | Seconds | 35-55 seconds |

In AgentCore, **you are responsible for managing session IDs**. The platform gives you full control over the session lifecycle, but that control comes with the responsibility of using it correctly. If you do not actively reuse session IDs, you will never get warm invocations.

## Cold Start Deep Dive: What Actually Happens in Those 40 Seconds?

When a cold start is triggered, AgentCore executes a seven-stage instantiation process before your request can be processed. Understanding each stage helps you know where the time goes and, critically, which stages you can optimize.

### The Seven Stages

**Stage 1: Authentication and Routing** *(Platform-managed)*

The Control Plane validates your AWS credentials using SigV4 or OAuth 2.0 and determines which runtime should handle your request. This stage is fast and contributes minimally to cold start time.

**Stage 2: Resource Allocation** *(Platform-managed)*

The platform finds an available host with sufficient capacity, reserves 2 vCPU cores and 8 GB of RAM for your new VM, and prepares the networking configuration. The time this takes depends on current platform load and resource availability.

**Stage 3: MicroVM Boot** *(Platform-managed)*

Firecracker boots the microVM. This involves initializing the kernel, setting up the virtual network interface, and mounting the filesystem. Thanks to Firecracker's design, this stage is fast — typically completing in well under a second.

**Stage 4: Image Pull and Decompression** *(Platform-managed)* ⚠️

This is the single biggest bottleneck in the cold start process and the stage where you have the most indirect influence. The platform pulls your Docker container image from Amazon ECR and decompresses its layers. The time this takes is directly proportional to your image size. A 2 GB image can add 20-25 seconds to your cold start. A 400 MB image might add only 5-8 seconds. We will discuss image optimization strategies later in this post.

**Stage 5: Runtime Initialization** *(Your code)*

The Python 3.11 interpreter starts up inside the container. This stage is where your code's execution begins. The interpreter itself starts quickly, but the time grows if you have many modules being imported at the top level.

**Stage 6: Dependency Loading** *(Your code)*

Python imports your dependencies — LangGraph, LangChain, the Bedrock SDK, and any other libraries your agent uses. Heavy libraries like PyTorch or large ML frameworks can add significant time here. This is a stage you can meaningfully optimize through lazy loading and careful dependency management.

**Stage 7: Entrypoint Execution** *(Your code)*

Your agent's entrypoint code runs. This typically involves creating the LangGraph agent instance, compiling the agent graph, and performing any initialization logic you have written. Graph compilation in particular can be time-consuming and is a good candidate for caching.

### The Three-Phase Timing Model

From a measurement perspective, these seven stages map to three phases that AgentCore tracks and publishes as CloudWatch metrics:

**Phase 1 — Firecracker VM Init** (Stages 1-4)
This is the platform-managed portion. You cannot instrument it directly from your runtime code. It is calculated as the remainder after subtracting the runtime and request phases from the total. Typical duration: 15-25 seconds, representing approximately 40% of total cold start time.

**Phase 2 — Runtime Startup** (Stages 5-7)
This is the portion your code controls. It can be instrumented with OpenTelemetry and measured via the RuntimeStartup span. Typical duration: 10-15 seconds, representing approximately 30% of total cold start time.

**Phase 3 — First Request Processing** (Request handling)
The actual processing of your first request, including payload deserialization, agent execution, LLM API calls, and response generation. Typical duration: 10-15 seconds, representing approximately 30% of total cold start time.

```
Total Cold Start = Phase 1 + Phase 2 + Phase 3
```

| Phase | What It Covers | Typical Duration | % of Total | Can You Optimize? |
|-------|---------------|-----------------|------------|-------------------|
| VM Init | Stages 1-4 (Platform) | 15-25 seconds | ~40% | Indirectly (image size) |
| Runtime Startup | Stages 5-7 (Your code) | 10-15 seconds | ~30% | Yes (imports, graph compile) |
| First Request | Request processing | 10-15 seconds | ~30% | Yes (prompts, caching) |
| **Total** | **End-to-end** | **35-55 seconds** | **100%** | **Partially** |

### What Triggers a Cold Start?

Cold starts are triggered in four situations:

1. **No session ID provided**: You make a request without including a session ID. AgentCore creates a new one and spins up a new VM.

2. **New session ID**: You provide a session ID that AgentCore has never seen before. A new VM is allocated.

3. **Session idle timeout**: Your session has been inactive for more than 15 minutes. The VM was terminated, and the next request with that session ID triggers a new cold start.

4. **Explicit session termination**: The session was programmatically terminated, and a subsequent request triggers a new VM allocation.

## Warm Start: What Happens When You Get It Right

When you correctly reuse a session ID within the 15-minute window, the experience is dramatically different. The Control Plane looks up your session ID, finds the running VM, and routes your request directly to it. None of the seven stages execute. The VM is already running, the Python interpreter is already loaded, your dependencies are already imported, and your agent graph is already compiled.

The only work that happens is processing your specific request: deserializing the payload, running the agent reasoning loop, making LLM API calls, executing any tool calls, and returning the response.

This is why warm invocations are so much faster. You are skipping 35-50 seconds of initialization work and going straight to the 2-8 seconds of actual request processing.

### What Gets Skipped in a Warm Start

When you reuse a session ID and the VM is still active, all of the following are skipped:

- ✅ VM resource allocation
- ✅ Firecracker VM boot
- ✅ Docker image pull from ECR
- ✅ Image layer decompression
- ✅ Python interpreter startup
- ✅ Dependency imports (LangGraph, LangChain, Bedrock SDK)
- ✅ Agent graph compilation
- ✅ Any custom initialization code in your entrypoint

Only the actual request processing happens. This is the steady-state performance of your agent, and it is what your users experience for all requests after the first one in a session.

## Session Lifecycle: A Complete Timeline

Let us walk through a concrete example of how a session evolves over time. This timeline illustrates exactly when cold and warm starts occur and why.

```
t = 0 seconds
Request 1 arrives with session_id = "session-abc-123" (NEW)
→ Control Plane: Session not found in registry
→ COLD START begins
→ Stage 1-7 execute sequentially
→ Response returned after ~40 seconds
→ Session "session-abc-123" now active in registry

t = 60 seconds
Request 2 arrives with session_id = "session-abc-123" (REUSE)
→ Control Plane: Session found, VM is running
→ WARM START
→ Request routed directly to existing VM
→ Response returned after ~4 seconds

t = 120 seconds
Request 3 arrives with session_id = "session-abc-123" (REUSE)
→ Control Plane: Session found, VM is running
→ WARM START
→ Response returned after ~3 seconds

t = 120 to t = 1020 seconds
No requests arrive for this session
→ 15-minute inactivity timer counting down
→ At t = 1020s, timer expires
→ VM terminated, session removed from registry

t = 1080 seconds
Request 4 arrives with session_id = "session-abc-123" (EXPIRED)
→ Control Plane: Session not found (was expired)
→ COLD START begins again
→ New VM allocated for same session ID
→ Response returned after ~40 seconds
```

This timeline illustrates a critical point: **the session ID itself does not guarantee a warm start**. What guarantees a warm start is the combination of reusing the same session ID AND making requests within the 15-minute inactivity window. If your application has periods of inactivity longer than 15 minutes, you should expect cold starts to occur and plan for them accordingly.

## Production-Ready Session Management in Python

Now that you understand the theory, let us look at how to implement session management correctly in Python. The following `SessionManager` class is the pattern we recommend to customers during AWS Professional Services engagements.

### The SessionManager Class

```python
import uuid
import time
import logging
from datetime import datetime
from typing import Optional

logger = logging.getLogger(__name__)

class SessionManager:
    """
    Production-ready session manager for AWS Bedrock AgentCore.
    
    Manages session ID lifecycle to maximize warm starts
    and minimize cold starts in production applications.
    
    Key behaviors:
    - create_session(): Forces a cold start (new UUID)
    - get_warm_session(): Returns existing warm session
    - record_invocation(): Keeps session alive (resets TTL)
    - is_session_warm(): Checks if session is within 15-min window
    - cleanup_expired_sessions(): Removes stale tracking data
    """
    
    # 15-minute TTL with 60-second safety buffer
    SESSION_TTL_SECONDS = 840  # 14 minutes
    
    def __init__(self):
        # Maps runtime_id -> list of session IDs
        self.sessions: dict = {}
        # Maps session_id -> last invocation timestamp
        self.session_last_used: dict = {}
        
    def create_session(self, runtime_id: str) -> str:
        """
        Create a new session ID.
        
        This ALWAYS results in a cold start because
        the session ID has never been seen by AgentCore.
        
        Use this for:
        - Cold start performance testing
        - Starting a genuinely new user session
        - After detecting a session has expired
        
        Args:
            runtime_id: The AgentCore runtime identifier
            
        Returns:
            New UUID session ID string
        """
        session_id = str(uuid.uuid4())
        
        if runtime_id not in self.sessions:
            self.sessions[runtime_id] = []
            
        self.sessions[runtime_id].append(session_id)
        self.session_last_used[session_id] = datetime.now()
        
        logger.info(
            f"Created new session {session_id} "
            f"for runtime {runtime_id} — cold start expected"
        )
        
        return session_id
    
    def get_warm_session(
        self, 
        runtime_id: str
    ) -> Optional[str]:
        """
        Get an existing warm session ID for a runtime.
        
        Returns a session ID that is still within the
        15-minute activity window, guaranteeing a warm start.
        
        Returns None if no warm session is available,
        in which case the caller should create a new session.
        
        Args:
            runtime_id: The AgentCore runtime identifier
            
        Returns:
            Existing warm session ID, or None
        """
        sessions = self.sessions.get(runtime_id, [])
        
        for session_id in sessions:
            if self.is_session_warm(session_id):
                logger.debug(
                    f"Found warm session {session_id} "
                    f"for runtime {runtime_id}"
                )
                return session_id
                
        logger.info(
            f"No warm session available for runtime "
            f"{runtime_id} — cold start will occur"
        )
        return None
    
    def is_session_warm(self, session_id: str) -> bool:
        """
        Check whether a session is still within
        the active window (less than 14 minutes old).
        
        We use 14 minutes (not 15) as a safety buffer
        to avoid race conditions near the TTL boundary.
        
        Args:
            session_id: The session ID to check
            
        Returns:
            True if session is warm, False if expired
        """
        if session_id not in self.session_last_used:
            return False
            
        last_used = self.session_last_used[session_id]
        age_seconds = (
            datetime.now() - last_used
        ).total_seconds()
        
        is_warm = age_seconds < self.SESSION_TTL_SECONDS
        
        if not is_warm:
            logger.debug(
                f"Session {session_id} has expired "
                f"(age: {age_seconds:.0f}s)"
            )
            
        return is_warm
    
    def record_invocation(self, session_id: str):
        """
        Record that a session was successfully used.
        
        Call this after every successful AgentCore invocation
        to reset the inactivity timer and keep the session warm.
        
        Failing to call this means your local tracking will
        diverge from the actual session state in AgentCore.
        
        Args:
            session_id: The session ID that was used
        """
        self.session_last_used[session_id] = datetime.now()
        logger.debug(f"Recorded invocation for session {session_id}")
    
    def cleanup_expired_sessions(self, runtime_id: str):
        """
        Remove expired sessions from local tracking.
        
        Call this periodically in long-running applications
        to prevent memory growth from accumulated expired sessions.
        
        Args:
            runtime_id: The AgentCore runtime identifier
        """
        sessions = self.sessions.get(runtime_id, [])
        active_sessions = []
        removed_count = 0
        
        for session_id in sessions:
            if self.is_session_warm(session_id):
                active_sessions.append(session_id)
            else:
                # Remove from last_used tracking
                self.session_last_used.pop(session_id, None)
                removed_count += 1
                
        self.sessions[runtime_id] = active_sessions
        
        if removed_count > 0:
            logger.info(
                f"Cleaned up {removed_count} expired sessions "
                f"for runtime {runtime_id}"
            )
```

### Using SessionManager in a Production Application

Here is how you would use the `SessionManager` in a real application. This example shows a credit memo generation service that correctly handles both the initial cold start and all subsequent warm invocations:

```python
import time
import logging
from datetime import datetime

logger = logging.getLogger(__name__)

class CreditMemoService:
    """
    Example production service using AgentCore
    with correct session management.
    """
    
    def __init__(self, agentcore_client, runtime_id: str):
        self.client = agentcore_client
        self.runtime_id = runtime_id
        self.session_manager = SessionManager()
        
    def generate_credit_memo(
        self, 
        company_id: str,
        invoice_data: dict
    ) -> dict:
        """
        Generate a credit memo using AgentCore.
        
        Automatically handles session reuse for warm starts.
        Falls back to cold start if no warm session available.
        """
        # Step 1: Try to get an existing warm session
        session_id = self.session_manager.get_warm_session(
            self.runtime_id
        )
        
        is_cold_start = session_id is None
        
        # Step 2: If no warm session, create a new one
        # (this will trigger a cold start)
        if session_id is None:
            session_id = self.session_manager.create_session(
                self.runtime_id
            )
            logger.info(
                f"No warm session available — "
                f"cold start expected for company {company_id}"
            )
        else:
            logger.info(
                f"Reusing warm session {session_id} "
                f"for company {company_id}"
            )
        
        # Step 3: Make the AgentCore request
        start_time = datetime.now()
        
        try:
            response = self.client.invoke(
                runtime_id=self.runtime_id,
                payload={
                    "query": (
                        f"Generate credit memo for "
                        f"company {company_id}"
                    ),
                    "invoice_data": invoice_data
                },
                session_id=session_id
            )
            
            duration_ms = (
                datetime.now() - start_time
            ).total_seconds() * 1000
            
            # Step 4: Record the successful invocation
            # This resets the 15-minute inactivity timer
            self.session_manager.record_invocation(session_id)
            
            logger.info(
                f"Request completed in {duration_ms:.0f}ms "
                f"({'cold' if is_cold_start else 'warm'} start)"
            )
            
            return response
            
        except Exception as e:
            logger.error(
                f"AgentCore invocation failed: {e}"
            )
            raise
```

### Handling Session Expiration Gracefully

In production, sessions will expire. Your application needs to handle this gracefully rather than failing:

```python
class ResilientCreditMemoService(CreditMemoService):
    """
    Extended service with automatic session recovery.
    Handles expired sessions without surfacing errors
    to the caller.
    """
    
    MAX_RETRIES = 3
    
    def generate_credit_memo_with_recovery(
        self,
        company_id: str,
        invoice_data: dict
    ) -> dict:
        """
        Generate credit memo with automatic session recovery.
        
        If the session has expired on the AgentCore side
        (but our local tracking hasn't caught up yet),
        this method automatically creates a new session
        and retries the request.
        """
        last_exception = None
        
        for attempt in range(self.MAX_RETRIES):
            try:
                return self.generate_credit_memo(
                    company_id=company_id,
                    invoice_data=invoice_data
                )
                
            except SessionExpiredException as e:
                logger.warning(
                    f"Session expired on attempt {attempt + 1}, "
                    f"creating new session and retrying"
                )
                last_exception = e
                
                # Force creation of a new session
                # on the next attempt
                self.session_manager.cleanup_expired_sessions(
                    self.runtime_id
                )
                
                # Exponential backoff between retries
                if attempt < self.MAX_RETRIES - 1:
                    wait_seconds = 2 ** attempt
                    time.sleep(wait_seconds)
                    
        raise last_exception
```

## Common Mistakes and How to Avoid Them

Based on our experience working with customers on AgentCore deployments, here are the four most common session management mistakes and how to fix them:

### Mistake 1: Creating a New Session ID for Every Request

**What it looks like:**
```python
# ❌ WRONG — This causes a cold start on every single request
def invoke_agent(query: str) -> dict:
    session_id = str(uuid.uuid4())  # New ID every time!
    return client.invoke(
        payload={"query": query},
        session_id=session_id
    )
```

**Why it happens:** Developers often generate a UUID as a "unique request identifier" without realizing that in AgentCore, the session ID controls VM routing, not just request tracking.

**The fix:**
```python
# ✅ CORRECT — Create once, reuse for subsequent requests
class AgentService:
    def __init__(self):
        self.session_manager = SessionManager()
        self.runtime_id = "my-agent-runtime"
    
    def invoke_agent(self, query: str) -> dict:
        # Get warm session or create new one
        session_id = (
            self.session_manager.get_warm_session(
                self.runtime_id
            ) 
            or 
            self.session_manager.create_session(
                self.runtime_id
            )
        )
        
        response = client.invoke(
            payload={"query": query},
            session_id=session_id
        )
        
        self.session_manager.record_invocation(session_id)
        return response
```

### Mistake 2: Not Saving the Session ID from the First Response

**What it looks like:**
```python
# ❌ WRONG — Session ID discarded after first request
response = client.invoke(payload={"query": "Hello"})
# session_id not saved — next request will be cold start
```

**The fix:**
```python
# ✅ CORRECT — Always capture and store the session ID
response = client.invoke(payload={"query": "Hello"})
session_id = response.get("session_id")
# Store session_id for reuse in subsequent requests
```

### Mistake 3: Ignoring the 15-Minute Timeout in Long-Running Tests

**What it looks like:**
Running a performance test with 100 iterations but with enough delay between iterations that the session expires partway through. The result is a mix of cold and warm measurements that produces misleading statistics.

**The fix:**
```python
# ✅ CORRECT — Track session age and refresh before expiry
for i in range(iterations):
    # Check if session is still warm before each request
    if not session_manager.is_session_warm(session_id):
        logger.warning(
            "Session expired during test — "
            "creating new session"
        )
        session_id = session_manager.create_session(
            runtime_id
        )
        # Note: This iteration will be a cold start
        # Consider excluding it from warm metrics
    
    response = client.invoke(
        payload={"query": f"Test query {i}"},
        session_id=session_id
    )
    session_manager.record_invocation(session_id)
```

### Mistake 4: Mixing Cold and Warm Measurements

**What it looks like:**
Including the first request (cold start) in warm invocation statistics, which inflates p99 and mean latency figures.

**The fix:**
Always run cold start tests and warm invocation tests separately, using different session management strategies for each. We will cover the complete measurement methodology in Part 2 of this series.

## Session Management Best Practices Checklist

## Optimization Strategies: Reducing Cold Start Time

While you cannot eliminate cold starts entirely, you can significantly reduce how long they take. Here are the most impactful optimizations, organized by the phase they affect.

### Phase 1 Optimization: Reduce Docker Image Size

The image pull stage (Stage 4) is the biggest variable in cold start time and the one where you have the most leverage. Every megabyte you remove from your Docker image directly reduces cold start time.

**Use multi-stage builds:**
```dockerfile
# Stage 1: Build environment
FROM python:3.11 AS builder
WORKDIR /build
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt \
    --target /build/packages

# Stage 2: Runtime environment (much smaller)
FROM python:3.11-slim AS runtime
WORKDIR /app

# Copy only the installed packages, not the build tools
COPY --from=builder /build/packages /app/packages
COPY src/ /app/src/

ENV PYTHONPATH=/app/packages
CMD ["python", "src/agent.py"]
```

**Remove unnecessary dependencies:**
Review your `requirements.txt` and ask whether each dependency is truly needed at runtime. Development tools, testing frameworks, and documentation generators have no place in a production container image.

**Use slim base images:**
Replace `python:3.11` (approximately 900 MB) with `python:3.11-slim` (approximately 130 MB) as your base image. This alone can reduce your image size by 700+ MB.

### Phase 2 Optimization: Lazy Load Heavy Libraries

Instead of importing all dependencies at module load time, defer imports until they are actually needed:

```python
# ❌ SLOW — All imports happen at startup
import torch
import transformers
from langchain.embeddings import HuggingFaceEmbeddings

def process_request(query: str):
    embeddings = HuggingFaceEmbeddings()
    # ...

# ✅ FAST — Heavy imports deferred until first use
def process_request(query: str):
    # Import only when needed
    from langchain.embeddings import HuggingFaceEmbeddings
    embeddings = HuggingFaceEmbeddings()
    # ...
```

## Summary

In this post, we covered the foundational concepts you need to understand and control AgentCore performance:

- **Firecracker microVMs** provide fast, secure, isolated execution environments for each AgentCore session, with sub-125ms boot times and strong multi-tenant isolation

- **The four-layer architecture** (Control Plane → Session → MicroVM → Container) determines what happens during every request

- **Session IDs are routing keys**: one session ID maps to exactly one Firecracker VM, and this mapping is the mechanism that determines whether you get a cold start or a warm start

- **Cold starts involve seven stages** and typically take 35-55 seconds. The image pull stage is the biggest bottleneck and the most impactful optimization target

- **Warm starts skip all initialization** and typically complete in 2-8 seconds, but they require you to explicitly reuse session IDs within the 15-minute inactivity window

- **AgentCore warm state is explicit**, unlike AWS Lambda where warm state is automatic. You are responsible for managing session IDs correctly

- **The SessionManager pattern** provides a production-ready approach to session lifecycle management that maximizes warm starts and handles expiration gracefully

- **Common mistakes** — especially creating new session IDs for every request — can make every invocation a cold start, resulting in 10-15x slower performance than necessary
